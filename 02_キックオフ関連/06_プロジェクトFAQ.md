# プロジェクトFAQ（想定質疑応答集）

## 📋 このFAQの使い方

キックオフやプロジェクト進行中に、業務チームから出そうな質問とその回答をまとめています。
事前に目を通しておくことで、スムーズな説明が可能になります。

---

## 🎯 プロジェクト全般について

### Q1. なぜ今このプロジェクトをやるのですか？

**A:** 3つの理由があります。

1. **すでに評価の仕組みがある**  
   会話ログと5段階評価を記録する基盤（Dataverse）が整っているため、データに基づいた改善ができます。

2. **ナレッジの散在が深刻化している**  
   掲示板のFAQが古いまま放置され、どれが最新かわからない状況です。このままではAIを入れても効果が出ません。

3. **問い合わせ対応の負担増加**  
   業務チームの業務を楽にするため、今が最適なタイミングです。

---

### Q2. このプロジェクトで私たちの負担は増えませんか？

**A:** **むしろ将来の負担を減らすための投資**です。

**最初の1-2ヶ月:**
- 週1回30分の定例会参加
- トピック定義シートの記入（数時間）
- ナレッジ整理（随時）

**3ヶ月目以降:**
- 問い合わせ件数が減少
- 「またこの質問か…」が激減
- 自動で対応できる範囲が拡大

**実際の時間投資:**
- 1ヶ月目: 週2-3時間程度
- 2ヶ月目: 週1-2時間程度
- 3ヶ月目以降: 週30分程度（定例会のみ）

---

### Q3. AIに詳しくないのですが、大丈夫ですか？

**A:** **大丈夫です。技術的なことは開発側が担当します。**

**業務チームに求めること:**
- 「この回答は正しいか？」の判断（✅ or ❌）
- 「この情報は古いか？最新か？」の判断
- 「ユーザーはこう聞いてくる」という業務知識

**開発チームがやること:**
- Copilot Studioの設定
- プロンプトの調整
- システムへの実装

つまり、業務チームは**「先生役」**、開発チームは**「実装担当」**です。

## 📝 トピック定義シートについて

### Q4. トピック定義シートの記入が難しいです。どうすればいいですか？

**A:** **完璧に埋める必要はありません。**

**最低限必要なのは:**
- ②ユーザーの質問（どんな言葉で聞かれるか）
- ⑤参照すべきナレッジ（根拠となる規程やマニュアル）

**③確認事項が思いつかない場合:**
→ 空欄でOKです。その場合、生成AI（自動回答）で対応します。

**記入に困ったら:**
- 「新人の〇〇さんにこの質問をされたら、何を聞き返していますか？」と自問してください
- それでもわからなければ、開発側に相談してください（一緒に整理します）

---

### Q5. 「ユーザーの質問」にどこまで書けばいいですか？

**A:** **思いつく限りの言い方を全て書いてください。**

❌ **悪い例:**
```
出張旅費申請
```

✅ **良い例:**
```
・出張旅費申請したい
・出張の交通費はどうする？
・新幹線代の精算
・宿泊費の領収書
・出張申請
・旅費精算
```

**ポイント:**
- 正式名称だけでなく、略語や話し言葉も書く
- 「PC」「パソコン」「パーソナルコンピューター」のように、言い方の違いも全て書く
- ユーザーの誤字も想定する（「有休」「有給」「年休」など）

---

### Q6. 「参照すべきナレッジ」がない場合はどうすればいいですか？

**A:** **正直に「マニュアルなし（口頭のみ）」と書いてください。**

これは問題ではなく、**「ナレッジ不足の可視化」という重要な気づき**です。

**その後の対応:**
1. 新規マニュアルを作成する（推奨）
2. 既存の規程に追記する
3. 「総務にお問い合わせください」と案内する設定にする

プロジェクトを通じて、「本当は文書化すべきだったもの」が明確になります。

## 🔍 ナレッジ整理について

### Q7. 掲示板の古いFAQを削除していいか不安です。

**A:** **削除ではなく、まずは「注記」から始めてください。**

**ステップ1: 注記を追加**
```
【古い情報です】
最新情報は「〇〇規程 第〇条」を参照してください。
```

**ステップ2: 様子を見る（1-2週間）**
→ 問い合わせがなければ削除

**ステップ3: バックアップを取ってから削除**
→ 念のため、削除した内容を別ファイルに保存

**削除判断のポイント:**
- 正式な規程と内容が**矛盾**している → 削除
- 正式な規程と内容が**同じ** → 削除（重複）
- 掲示板にしかない**有用な補足** → 正式版に統合

---

### Q8. 「正式版」を決める権限がありません。

**A:** **プロジェクトで承認プロセスを明確にしましょう。**

**推奨フロー:**
```
業務チームが提案 → 関係部署と合意 → 正式版として登録
```

**関係部署の例:**
- 経費精算 → 経理部と合意
- 勤怠 → 人事部と合意
- 福利厚生 → 人事部と合意

**キックオフ時に決めておくこと:**
- 誰が最終承認するか？
- どのくらいの頻度でレビューするか？

## 🤖 AIエージェントの仕組みについて

### Q9. AIは完璧に答えられるようになりますか？

**A:** **100%は不可能ですが、70-80%の自己解決率は十分達成可能です。**

**AIが得意なこと:**
- 規程やマニュアルに書いてある内容の照会
- 頻度の高い定型的な質問
- 24時間365日対応

**AIが苦手なこと:**
- 個別事情の判断（「私の場合はどうなる?」）
- 複雑な例外処理
- 感情的なケア

**だからこそ:**
- AIで答えられるものはAIに任せる
- 人間にしかできないことに時間を使う

---

### Q10. AIが間違った回答をしたらどうなりますか？

**A:** **低評価がつき、定例会で検出されます。**

**仕組み:**
1. ユーザーが5段階評価で「⭐」や「⭐⭐」をつける
2. 定例会で低評価ログをレビュー
3. 原因を分析（ナレッジ不足？AI理解ミス？）
4. ナレッジを追加・修正、またはAIを調整

**つまり:**
- 間違いは「失敗」ではなく「改善の種」
- 継続的に精度が向上するサイクル

---

### Q11. 「トピック」と「生成AI」の違いは何ですか？

**A:** **「質問を聞き返すか？」で決まります。**

| 項目 | トピック（会話フロー） | 生成AI（自動回答） |
|:---|:---|:---|
| **使い分け** | 条件分岐が必要 | 単純な照会 |
| **具体例** | 「通勤ですか？出張ですか？」と聞く | 「有給の申請方法は？」→ 規程を引用 |
| **実装** | 開発側がフローを組む | ナレッジを登録するだけ |

**記入例:**
```
【トピックが必要】
質問: 交通費を精算したい
確認: 通勤か？出張か？領収書はあるか？
回答: 条件によって案内先が変わる

【生成AIで対応可能】
質問: 有給の申請方法は？
回答: 就業規則を引用して一意に答えられる
```

## 📊 評価・改善について

### Q12. 自己解決率70%は本当に達成できますか？

**A:** **段階的に達成します。**

**現実的な目標:**
- 1ヶ月目: 40-50%（立ち上げ期）
- 2ヶ月目: 55-65%（改善期）
- 3ヶ月目: 70%以上（安定期）

**達成のカギ:**
- よく来る質問（Top 10）を優先的に整備
- 低評価ログを毎週レビューして改善
- ナレッジの矛盾を解消

---

### Q13. 低評価がついたら、誰かが責められますか？

**A:** **いいえ。低評価は「改善のヒント」です。**

**定例会での扱い方:**
```
❌ 悪い例:
「このトピックを作った人が悪い」

✅ 良い例:
「このログを見ると、掲示板の古い情報を拾っていますね。
 正式版を決めて、掲示板の情報を削除しましょう」
```

**プロジェクトの文化:**
- 低評価は「宝物」（改善の材料）
- 失敗を責めない
- チーム全体で改善していく

## ⏰ 時間・スケジュールについて

### Q14. 週1回の定例会は必須ですか？

**A:** **はい。継続改善の要です。**

**定例会の重要性:**
- 低評価ログの早期発見
- ナレッジ整理の進捗確認
- チーム間の認識合わせ

**もし週1回が難しい場合:**
- 隔週（2週間に1回）でスタート
- ただし、改善スピードは半減します

**所要時間:**
- 標準: 30分
- 最大: 45分（議論が必要な場合）

---

### Q15. 3ヶ月で本当に完成しますか？

**A:** **「完成」ではなく「継続改善の仕組みができる」が正確です。**

**3ヶ月後の状態:**
- よく来る質問（Top 10）には答えられる
- 自己解決率70%達成
- 定例会が定着し、改善サイクルが回っている

**3ヶ月後も続くこと:**
- 新しい質問への対応（随時）
- 制度変更時のナレッジ更新
- 定例会（月1回程度に減らしてもOK）

つまり、3ヶ月は「立ち上げ期」で、その後は**「運用・継続改善期」**に移行します。

## 🛠️ 技術・システムについて

### Q16. Copilot Studioは難しくないですか？

**A:** **業務チームは使い方を覚える必要はありません。**

**業務チームがやること:**
- Copilot Studioの画面を見ながら動作確認
- 「この回答で良いか？」の判断

**開発チームがやること:**
- Copilot Studioの設定・実装
- トピックの作成
- プロンプトの調整

---

### Q17. Dataverseって何ですか？

**A:** **会話ログを保存するデータベースです。**

**業務チームが知っておくべきこと:**
- ユーザーの質問とAIの回答が記録される
- 5段階評価も記録される
- これらのデータを使って改善する

**業務チームが操作する必要はありません:**
- 開発側がPower BIで可視化します
- 定例会で「見やすい形」で提示されます

## 💼 業務への影響について

### Q18. 通常業務に影響は出ませんか？

**A:** **最初の1-2ヶ月は若干の時間投資が必要です。**

**時間配分の目安:**
| 時期 | 週あたり時間 | 主な作業 |
|:---|:---|:---|
| 1ヶ月目 | 2-3時間 | トピック定義シート記入、ナレッジ整理 |
| 2ヶ月目 | 1-2時間 | 定例会、ナレッジ追加 |
| 3ヶ月目以降 | 30分-1時間 | 定例会のみ |

**ROI（投資対効果）:**
- 投資: 最初の2ヶ月で約20時間
- リターン: 問い合わせ対応が30%削減（月10-20時間削減）
- 回収期間: 約2-3ヶ月

---

### Q19. プロジェクト終了後、誰がメンテナンスしますか？

**A:** **業務チームが日常業務の一部として継続します。**

**定常運用時の作業（月1-2時間程度）:**
- 月1回の定例会（30分）
- 制度変更時のナレッジ更新（随時）

## 🎯 成果・効果について

### Q20. 本当に問い合わせは減りますか？

**A:** **実績ベースで30-50%削減が可能です。**

**削減できるもの:**
- 「マニュアルを見れば書いてある」質問
- 定型的な手続き案内
- 営業時間外の質問（AIが24時間対応）

**削減できないもの:**
- 個別事情の相談
- 複雑な例外処理
- システムトラブルの問い合わせ

**効果測定:**
- AI導入前の3ヶ月間の問い合わせ数を記録
- 導入後の問い合わせ数と比較
- KPIダッシュボードで可視化

**A:** **いつでも相談してください。**

**連絡先:**
- プロジェクト管理: 〇〇（開発側）
- 業務窓口: 〇〇（業務側）

**よくある相談:**
- トピック定義シートの記入方法
- ナレッジ整理の判断
- 定例会で議論が進まない
- スケジュールの調整

**相談方法:**
- チャット（Teams/Slackなど）
- メール
- 定例会の場で
