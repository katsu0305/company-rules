# 全社展開時のロール定義とRACI

**目的**: 部門横展開を成功させるための役割分担と責任範囲の明確化

---

## ✅ 推奨ロール構成(5つ)

### 1. **本社IT統轄ロール(CoE: Center of Excellence)**

#### 役割
- プラットフォーム管理(環境構築、ライセンス管理)
- 全社標準・ガバナンスの策定
- 部門開発者のトレーニング・サポート
- 技術エスカレーション対応
- セキュリティ・監査
- ベストプラクティスの横展開

#### 主な業務
- Copilot Studio環境の構築・保守
- 部門開発者向けトレーニングプログラムの実施
- 技術的な質問への回答(週1回の相談会)
- 複雑なトピック実装のレビュー・サポート
- セキュリティポリシーの策定と監査
- 全社共有会の開催(四半期1回)

#### 必要なスキル
- Copilot Studio / Power Platform の専門知識
- プロジェクトマネジメント
- トレーニング・ファシリテーション
- ガバナンス・セキュリティ知識

#### 人数
2-3名(専任または兼務)

#### 稼働目安
- Phase 1(準備): 50%
- Phase 2(実装): 30%
- Phase 3(運用): 20%

---

### 2. **部門管理者ロール** ⭐

#### 役割
- 部門内のAI活用推進責任者
- プロジェクトの優先順位決定
- リソース配分(稼働時間の確保)
- 経営層への報告
- 投資判断(追加リソースの承認など)
- 部門内の意思決定

#### 主な業務
- キックオフへの参加と承認
- 部門開発者・業務担当者の選定と承認
- 月次進捗レビューと課題解決
- トピック優先順位の決定
- 予算確保と稼働時間の承認
- 経営層への効果報告

#### 必要なスキル
- 部門業務の理解
- リーダーシップ
- 意思決定力
- ステークホルダーマネジメント

#### 人数
各部門1名(部長・課長クラス)

#### 稼働目安
- Phase 1(準備): 10%
- Phase 2(実装): 10%
- Phase 3(運用): 5%

#### なぜ必要か
- 開発者や業務担当者だけでは意思決定できない場面がある
- 上からのコミットメントがないと優先順位が下がる
- 経営層と現場の橋渡し役が必要
- リソース確保の権限が必要

---

### 3. **部門開発者ロール(シチズンデベロッパー)**

#### 役割
- トピック設計・実装
- テスト・デプロイ
- 週次レビュー会のファシリテーション
- ログ分析と改善提案
- 業務担当者への技術サポート
- ドキュメント整備

#### 主な業務
- トピック定義シートに基づく実装
- テスト環境での動作確認
- 本番環境へのデプロイ申請
- 週次ログレビュー会の開催
- KPIモニタリングと改善提案
- 新規トピック候補の洗い出し
- 業務担当者からの相談対応

#### 必要なスキル
- 基本的なIT操作(Excel/Word)
- 論理的思考力
- 業務知識(3年以上推奨)
- 学習意欲
- コミュニケーション能力

#### 人数
各部門1-2名(メイン+サブ)

#### 稼働目安
- Phase 1(準備): 50%(トレーニング含む)
- Phase 2(実装): 80%
- Phase 3(運用): 30%

---

### 4. **業務担当者ロール(ナレッジオーナー)**

#### 役割
- よく来る質問の洗い出し
- トピック定義シートの記入
- ナレッジの整理・更新
- 回答内容の正確性チェック
- エンドユーザーからの個別問い合わせ対応(AI解決できない場合)

#### 主な業務
- 「よく来る質問Top 10」の特定
- トピック定義シートへの記入
- 既存ナレッジの棚卸しと整理
- AIエージェントの回答内容レビュー
- ナレッジの定期更新(月1回)
- 週次レビュー会への参加

#### 必要なスキル
- 部門業務の深い理解
- 文書作成能力
- ナレッジ整理能力
- ユーザー視点

#### 人数
各部門3-5名(ローテーション可)

#### 稼働目安
- Phase 1(準備): 30%
- Phase 2(実装): 50%
- Phase 3(運用): 20%

---

### 5. **エンドユーザーロール**

#### 役割
- AIエージェントの利用
- フィードバック(評価)の提供
- 解決できない場合のエスカレーション

#### 主な業務
- 日常業務でのAIエージェント利用
- 回答への5段階評価
- 低評価時のコメント記入
- AIで解決できない場合の業務担当者への問い合わせ

#### 必要なスキル
- 基本的なPC操作

#### 人数
全社員

#### 稼働目安
- 日常業務の中で随時

## 📊 ロール間の関係性

```
                    [経営層]
                       ↓ 承認・予算
    ┌─────────────────┼─────────────────┐
    ↓                  ↓                  ↓
[本社IT統轄(CoE)]  [部門管理者A]    [部門管理者B]
    ↓ トレーニング     ↓ 指示・支援      ↓ 指示・支援
    ↓ 技術サポート     ↓                ↓
    ↓            [部門開発者A] ←→ [業務担当者A]
    ↓                  ↓ 実装      ↓ ナレッジ提供
    ↓            [AIエージェント A]
    ↓                  ↓
    ↓            [エンドユーザー(部門A)]
    ↓                  ↓ フィードバック
    ↓            [部門開発者A] → 改善
    ↓
    └──────→ [部門開発者B] ←→ [業務担当者B]
                   ↓ 実装      ↓ ナレッジ提供
             [AIエージェント B]
                   ↓
             [エンドユーザー(部門B)]
                   ↓ フィードバック
             [部門開発者B] → 改善
```

**ポイント:**
- 本社IT統轄は全部門を横断的にサポート
- 各部門は独立して運用(自律性)
- 部門間で知見を共有(コミュニティ活動)

## 🎯 各ロールの責任範囲(RACI表)

| タスク | 本社IT | 部門管理者 | 部門開発者 | 業務担当者 | エンドユーザー |
|:---|:---:|:---:|:---:|:---:|:---:|
| **プラットフォーム管理** |
| 環境構築 | **R/A** | I | I | - | - |
| ライセンス管理 | **R/A** | I | - | - | - |
| セキュリティ監査 | **R/A** | I | C | - | - |
| 全社標準策定 | **R/A** | C | C | - | - |
| **トレーニング・育成** |
| トレーニング実施 | **R/A** | C | I | I | - |
| 技術サポート | **R/A** | I | C | C | - |
| 定例相談会 | **R/A** | I | C | - | - |
| **部門戦略・管理** |
| 部門戦略決定 | C | **R/A** | C | I | - |
| リソース配分 | I | **R/A** | I | I | - |
| 優先順位決定 | C | **R/A** | C | C | - |
| 経営報告 | C | **R/A** | C | - | - |
| 予算確保 | I | **R/A** | I | - | - |
| **トピック開発** |
| トピック設計 | C | I | **R/A** | C | - |
| トピック実装 | C | I | **R/A** | I | - |
| テスト実施 | C | I | **R/A** | C | - |
| デプロイ申請 | A | I | **R** | - | - |
| デプロイ承認 | **A** | I | R | - | - |
| **ナレッジ管理** |
| よく来る質問の洗い出し | - | I | C | **R/A** | I |
| トピック定義シート記入 | - | I | C | **R/A** | - |
| ナレッジ整理 | - | I | C | **R/A** | - |
| 内容正確性チェック | - | I | C | **R/A** | - |
| ナレッジ更新 | - | I | C | **R/A** | - |
| **運用・改善** |
| 週次レビュー会 | I | C | **R/A** | **A** | - |
| ログ分析 | C | I | **R/A** | C | - |
| 改善実施 | C | I | **R/A** | C | - |
| KPIモニタリング | C | A | **R** | I | - |
| **利用・フィードバック** |
| AIエージェント利用 | - | - | I | I | **R/A** |
| 評価・フィードバック | - | - | I | I | **R/A** |
| 問い合わせエスカレーション | - | - | I | **A** | **R** |

**凡例:**
- **R (Responsible)**: 実行責任者 - 実際に作業を行う人
- **A (Accountable)**: 説明責任者 - 最終決裁権を持つ人(各タスク1名のみ)
- **C (Consulted)**: 相談先 - 意見を求められる人
- **I (Informed)**: 報告先 - 結果を知らされる人

## 💡 部門管理者ロールの重要性

### シーン1: リソース確保

**❌ NG例(管理者不在):**
```
開発者: 「トピック作成の時間が欲しいです」
直属上司: 「通常業務を優先して。空いた時間でやって」
→ 進まない、優先順位が下がる
```

**✅ OK例(管理者あり):**
```
部門管理者: 「AI開発は部門の重点施策。週10時間は確保してください」
直属上司: 「承知しました。業務を調整します」
→ 進む、計画通りに実行できる
```

---

### シーン2: 優先順位の決定

**❌ NG例(管理者不在):**
```
開発者: 「A案(経費精算)とB案(休暇申請)、どちらを先に?」
業務担当者: 「両方重要です!」
→ 決まらない、両方中途半端
```

**✅ OK例(管理者あり):**
```
部門管理者: 「KPIへのインパクトと緊急性からA案を優先」
→ 明確、集中して取り組める
```

---

### シーン3: 経営層への報告

**❌ NG例(管理者不在):**
```
開発者が直接経営層に報告
→ 部分最適の視点、説得力が弱い、継続投資が得られない
```

**✅ OK例(管理者あり):**
```
部門管理者が部門全体の成果として報告
→ 全体最適、予算確保につながる、組織としての取り組みと認識される
```

---

### シーン4: 組織横断の調整

**❌ NG例(管理者不在):**
```
開発者: 「人事システムのデータが必要です」
情報システム部: 「部門レベルでの依頼では対応できません」
→ 進まない
```

**✅ OK例(管理者あり):**
```
部門管理者: 「部門の公式プロジェクトとして依頼します」
情報システム部: 「承知しました。調整します」
→ 進む
```

## ⚠️ よくある失敗パターンと対策

### 失敗例1: 部門管理者が不在

**症状:**
- 開発者が孤立し、相談先がない
- リソース不足で進まない、他業務に押される
- 優先順位が曖昧で、方向性が定まらない
- 経営層への報告がなく、認知されない

**影響:**
- プロジェクトが停滞
- 開発者のモチベーション低下
- 投資対効果が不明確

**対策:**
- 部門管理者を明確に任命(部長・課長クラス)
- 月1回の定例報告を義務化
- 管理者の役割をジョブディスクリプションに明記
- 人事評価項目に組み込む

---

### 失敗例2: 業務担当者が固定されすぎ

**症状:**
- 特定の人に負荷集中
- 属人化が進む
- 幅広い視点が得られない
- 担当者が異動すると運用が止まる

**影響:**
- 持続可能性が低い
- 偏った回答内容
- ナレッジの質の低下

**対策:**
- 3-5名でローテーション
- 月1回程度の関与でOK(負荷を分散)
- ドキュメント化を徹底
- 定期的なメンバー入れ替え

---

### 失敗例3: 本社ITが現場実装まで担当

**症状:**
- 本社ITがパンク、他業務が圧迫される
- 現場の業務理解が不足し、的外れな実装
- スケールしない(他部門展開できない)
- 現場の当事者意識が低い

**影響:**
- 持続不可能な体制
- 品質の低下
- 展開スピードが遅い

**対策:**
- 本社ITは技術支援に徹する
- 実装は部門開発者に任せる
- トレーニングを充実させる
- CoEとして標準化・ガバナンスに注力

---

### 失敗例4: エンドユーザーのフィードバックが集まらない

**症状:**
- 評価が少ない、コメントが空欄
- 改善の方向性が不明
- 実際の利用状況が見えない

**影響:**
- 改善が進まない
- ユーザーニーズとのギャップ
- KPI達成が困難

**対策:**
- フィードバックの重要性を啓蒙
- 評価必須の仕組み化
- 定期的なアンケート実施
- フィードバックへの対応を可視化

---

### 失敗例5: 部門開発者が1名のみ

**症状:**
- 休暇・病欠時に対応できない
- 属人化が進む
- 負荷が高すぎて離脱

**影響:**
- 運用停止リスク
- 開発者の燃え尽き
- ナレッジの消失

**対策:**
- 必ず2名体制(メイン+サブ)
- ドキュメント整備を徹底
- 定期的なナレッジ共有

## 🔄 ロール間の連携フロー

### トピック追加の標準フロー

```
1. [エンドユーザー] 
   → AIで解決できない質問が多発
   ↓
2. [業務担当者]
   → ログを確認、新規トピック候補を特定
   → トピック定義シートに記入
   ↓
3. [部門開発者]
   → シートをレビュー、不明点を業務担当者に確認
   → テスト環境で実装
   → 業務担当者に動作確認依頼
   ↓
4. [業務担当者]
   → 内容チェック、フィードバック
   ↓
5. [部門開発者]
   → 修正・調整
   → 本番デプロイ申請
   ↓
6. [本社IT統轄]
   → レビュー・承認
   → 本番環境にデプロイ
   ↓
7. [エンドユーザー]
   → 新トピック利用開始
   → フィードバック提供
   ↓
8. [部門開発者]
   → ログ確認、継続改善
```

---

### 課題エスカレーションフロー

```
1. [部門開発者]
   → 技術的な問題に直面
   ↓
2. [本社IT統轄]
   → 技術サポート
   → 解決しない場合、ベンダーへエスカレーション
   
または

1. [部門開発者]
   → リソース・優先順位の問題に直面
   ↓
2. [部門管理者]
   → 意思決定・調整
   → 解決しない場合、経営層へエスカレーション
```

## 💼 組織横断の調整が必要な場面

### ケース1: データ連携が必要

**関係者:**
- 部門開発者(依頼元)
- 情報システム部(データ提供側)

**調整フロー:**
1. 部門開発者が必要性を特定
2. 部門管理者が正式依頼
3. 本社IT統轄が技術的な調整
4. 情報システム部が実装

---

### ケース2: 全社ポリシーとの調整

**関係者:**
- 部門開発者(実装者)
- 本社IT統轄(ガバナンス管理)
- 法務・コンプライアンス部門
- 部門管理者(承認)

**調整フロー:**
1. 部門開発者がポリシー疑義を発見
2. 本社IT統轄に相談
3. 必要に応じて法務・コンプライアンスに確認
4. 方針決定・実装

## ✅ まとめ

### 成功の鍵

1. **明確なロール定義**: 誰が何をするか明確
2. **部門管理者の関与**: 意思決定・リソース確保
3. **2名体制**: 属人化防止、継続性確保
4. **定期的なコミュニケーション**: 週次・月次・四半期
5. **本社IT統轄のサポート**: 技術面・ガバナンス面

---

### 避けるべき落とし穴

1. ❌ 部門管理者不在 → リソース確保できず停滞
2. ❌ 開発者1名体制 → 属人化・運用停止リスク
3. ❌ 本社ITの抱え込み → スケールしない
4. ❌ 業務担当者固定 → 偏り・負荷集中
5. ❌ フィードバック不足 → 改善が進まない

---

### 理想的な体制

```
【各部門】
- 部門管理者: 1名(部長・課長クラス)
- 部門開発者: 2名(メイン+サブ)
- 業務担当者: 3-5名(ローテーション)
- エンドユーザー: 全員

【全社】
- 本社IT統轄(CoE): 2-3名
```

この体制で、**持続可能で自律的な全社AI活用**が実現できます。
